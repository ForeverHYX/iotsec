# Week 7: IoT 与 IoT 安全

## 零基础导读

IoT 可以先用一个贯穿例子理解：家里的温湿度传感器每分钟测一次环境，感知层把物理量变成数字数据，网络层通过 Wi-Fi、ZigBee、LoRa、NB-IoT、5G 或 LPWAN 把数据送出去，管理层在边缘设备或云端存储、清洗、分析，应用层根据规则或模型决定是否打开空调、报警或给用户推送通知。这个链条里任何一段出错，都会影响真实世界动作。

IoT 不能直接照搬传统互联网安全，原因是设备数量多、类型杂、资源弱、生命周期长。很多节点是电池供电，通信芯片是能耗大户；有些设备没有屏幕键盘，不适合输入复杂密码；固件更新不稳定，部署后可能十年不换；企业里还有 Shadow IoT，也就是没有被 IT 资产管理发现的摄像头、打印机、HVAC、VoIP 电话。传统服务器可以集中打补丁，IoT 往往分散在物理空间里，甚至被攻击后会影响门锁、医疗、交通或工业控制。

本章后半的语音案例属于感知安全。语音助手或说话人识别系统通常把音频转成特征或 embedding，再由 scorer 判断是不是某个人或某条命令。对抗样本是在声音里加入扰动，让模型误判；语音克隆是合成听起来像目标人的声音。这说明 IoT 安全不只关心网络包，还关心传感器输入是否可信、AI 模型是否能被物理世界的信号欺骗。

管理层之所以重要，是因为 IoT 数据通常不会只停在设备本地。网关或云平台要做数据接入、存储、索引、分析、告警和设备管理。存储形态可包括 DAS/NAS/SAN：DAS 是直接挂在服务器上的存储，NAS 是通过文件服务共享的网络存储，SAN 是面向块设备的专用存储网络。云计算服务可按 IaaS/PaaS/SaaS 理解：IaaS 提供虚拟机、网络和磁盘，PaaS 提供数据库、消息队列、函数计算等平台能力，SaaS 则是直接可用的业务应用。安全上要问：数据存在哪里、谁能访问、日志是否可审计、云 API 是否越权、设备证书和密钥如何轮换。

## 本章知识地图

1. **完整数据流**：物理对象 -> 传感器采集 -> 网络传输 -> 云/边缘处理 -> 应用决策 -> 执行器改变现实世界。
2. **IoT 四层**：感知层、网络层、管理层、应用层；每层都有不同资产和攻击面。
3. **技术背景**：WSN、CPS、AIoT、LPWAN、LoRa、NB-IoT、Matter、edge device、cloud/fog 都是 IoT 生态的一部分。
4. **管理层与云**：DAS/NAS/SAN 解决数据存储形态，IaaS/PaaS/SaaS 解决云服务抽象层次；它们共同决定 IoT 数据如何被保存、处理、审计和恢复。
5. **安全需求**：接入安全、通信安全、数据隐私、计算系统安全、感知安全、服务质量与恢复。
6. **语音感知安全**：speaker recognition -> embedding/scorer -> 对抗扰动或语音克隆 -> 认证和命令执行被欺骗。

## 初学者常见疑问

问：IoT 四层和 Security Architecture 是一回事吗？

答：四层结构描述系统怎么组成，Security Architecture 描述每层要怎么保护。感知层要防假传感器和假数据；网络层要加密、认证、抗重放、抗 DoS；管理层要保护云/边缘平台、数据库和设备管理；应用层要保护业务逻辑和用户权限。它们对应但不完全相同，因为审计、密钥管理、隐私治理是跨层能力。

问：什么是 Shadow IoT，为什么危险？

答：Shadow IoT 是没有进入正式 IT 资产清单的联网设备，例如部门自己买的摄像头、会议设备、智能灯、打印机。安全团队不知道它存在，就不会给它改默认口令、升级固件、监控异常流量。攻击者常把这些弱设备当入口，再横向移动到更重要的网络。

问：为什么语音攻击算 IoT 安全？

答：IoT 把传感器输入直接接到应用决策。语音锁、智能音箱、移动银行语音认证都把声音当身份或命令。如果攻击者用语音克隆冒充主人，或用对抗扰动让模型把一句话识别成另一条命令，问题发生在感知层，但后果会进入应用层和物理世界。

## 1. 本章速览

本章先介绍 IoT 的历史、体系结构和核心技术，再分析 IoT 为什么成为安全重灾区，最后用语音助手、扬声器识别、对抗样本、语音克隆和去身份化作为 IoT 感知安全案例。考试重点是 IoT 四层架构、设备受限和大规模异构带来的安全挑战，以及“感知层、网络层、管理层、应用层”各自的安全需求。

## 2. IoT 的定义与来源

- **Internet of Things**：把物理世界中的物体通过传感、通信、计算与互联网连接，实现识别、定位、跟踪、监管和智能控制。
- 早期来源包括 RFID、无线传感器网络 WSN、普适计算、嵌入式系统。
- WSN 是重要前身：传感器节点集成传感器、微处理器、无线通信芯片和电源，能够采集、处理并传输信息。
- IPv6、LPWAN、LoRa、NB-IoT、5G、AIoT 等技术推动 IoT 大规模部署。

## 3. IoT 四层结构

- **感知层 Sensing Layer**：RFID、无线传感器、摄像头、麦克风、智能终端等产生数据，是物理世界和网络空间的接口。
- **网络层 Network Layer**：互联网、移动通信网、局域网、LPWAN 等负责数据传输与互联。
- **管理层 Management Layer**：数据存储、云计算、大数据、智能处理、设备管理，是 IoT “智慧”的来源。
- **应用层 Application Layer**：智能交通、物流、建筑、环境监测、工业控制、医疗、智能家居等具体场景。

## 4. IoT 节点硬件组成

- **传感器**：根据应用需求选择，可能需要 ADC、校准、特定采样方式。
- **微处理器/MCU**：负责计算、控制和本地处理，常集成存储、Flash、数字 IO、ADC。
- **通信芯片**：通常是节点能耗大户，发送和接收状态都会消耗显著电流。
- **能源模块**：电池部署方便但容量受环境和电压影响；太阳能等能量采集可延长寿命但带来随机性。

## 5. IoT 主要特征

- **大规模终端**：设备数量极大，传统人工管理方法失效。
- **感知普遍化**：物理对象持续生成数据。
- **异构性强**：设备、协议、算力、功耗、厂商、生命周期差异巨大。
- **智能处理**：机器学习、数据挖掘、专家系统等参与决策。
- **连接方式扩展**：从人联网扩展到物物互联、边缘计算、云端协同。
- **典型矛盾 Contradictions**：高性能 vs 低能耗，强安全/强隐私 vs 低算力/低存储，实时可用性 vs 低带宽，大规模开放互联 vs 可管理/可隔离，数据共享智能化 vs 数据最小化。

## 6. IoT 为什么是安全重灾区

- **攻击面巨大**：连接设备数量多，任何弱节点都可能成为入口。
- **设备不可见**：许多企业环境中存在非 IT 采购的 Shadow IoT，传统资产管理看不到。
- **默认口令和弱凭据**：仍是 IoT 入侵和僵尸网络的重要原因。
- **漏洞修复困难**：设备生命周期长、固件更新机制弱、厂商支持不稳定。
- **物理影响真实**：攻击不仅泄露数据，还可能控制门禁、摄像头、HVAC、能源、医疗设备等。
- **标准和管理滞后**：新设备、新协议、新场景快速出现，安全要求常滞后于部署。

## 7. 典型风险场景

- **摄像头/路由器僵尸网络**：被控制后用于 DDoS。
- **IP 安防系统**：私有无线协议缺少认证和加密，可能被远程控制。
- **HVAC/能源表**：可作为进入企业网络的入口，也可造成物理环境异常。
- **视频会议系统**：被接管后泄露屏幕、音视频和内部资料。
- **打印机**：网络打印设备可能泄露文档或成为横向移动跳板。
- **VoIP 电话**：配置错误可导致监听、伪造或网络入口。
- **智能灯泡/家居设备**：mesh/Wi-Fi 通信可被嗅探或控制。

## 8. IoT 安全需求

- **通信安全**：认证、加密、完整性、抗重放、抗 DoS。
- **接入安全**：设备、用户、网关和云服务在接入前要完成身份认证和授权，防止伪造节点、非法 reader、恶意 App 或默认口令设备进入系统。
- **数据隐私安全**：感知数据可能包含位置、健康、行为、习惯、身份等敏感信息，需要最小化采集、访问控制和隐私保护。
- **计算系统安全**：固件、OS、应用、云端、边缘节点都需要防漏洞、防恶意代码、防越权。
- **感知安全**：传感器输入可被欺骗，如对抗样本、伪造语音、物理扰动。
- **安全控制**：设备发现、身份管理、密钥管理、日志审计、更新与恢复。
- **服务质量与恢复**：IoT 常连接物理过程，DoS、延迟和错误控制会影响现实安全，因此还要考虑异常检测、降级运行、备份通道和安全恢复。
- **PPT demand 细项**：IoT 还要做 `trust management`，即在海量异构节点之间评估谁可信；接入安全要防 `malicious-node access`，即恶意节点伪装成合法传感器、网关或终端接入；传输安全要防 `monitoring/destruction of transmitted data`，也就是监听、破坏、删除或篡改正在传的数据；海量终端同时上线会造成 `congestion-driven DoS`；多源传感器和边缘/云同步不一致会带来 `poor data consistency`；数据外包到云后会出现 `outsourced-data privacy`，需要支持密文存储、`ciphertext retrieval/operation` 和最小暴露。

## 9. IoT 安全架构思路

- **感知安全**：保护传感器、RFID、摄像头、麦克风等输入源，防止假数据、伪造身份和物理篡改。
- **传输安全**：端到端或链路层加密、认证、完整性保护、密钥管理。
- **处理安全**：云/边缘平台访问控制、数据隔离、可信执行、异常检测。
- **管理层数据安全**：IoT 平台常把传感数据写入 DAS/NAS/SAN 或云存储，并通过 IaaS/PaaS/SaaS 交付计算和应用能力。风险包括多租户隔离失败、数据库权限过宽、对象存储公开、设备影子状态被篡改、日志缺失和备份不可恢复。
- **应用安全**：权限控制、业务逻辑保护、用户隐私、审计。
- **整体协同**：单层安全不能保证整体安全，跨层交互和组合漏洞必须考虑。
- **横向能力**：安全控制、审计、不可否认、隐私治理和密钥生命周期管理应跨越四层。例如感知层生成的数据在传输、处理和应用层都要保留访问控制与审计链路。
- **PPT architecture 术语**：感知层物理保护包括 `anti-detection`、`anti-radiation` 和 `physical secrecy`，分别对应防侦测、防辐射泄露和物理保密；网络层/管理层要用 `digital signatures`、加密、认证和访问控制；跨层还要依赖标准化 `security protocols`，否则每个设备厂商自造协议会导致不可互通和不可审计。

## 10. 语音助手与感知安全案例

- 智能音箱、手机、语音锁、移动银行语音密码等把语音作为身份和命令输入。
- **对抗样本攻击**：在原始语音上加入微小扰动，使说话人识别系统误判。
- **通用/可迁移扰动**：课件中的 PhoneyTalker/PhyTalker 思路试图生成可对未见模型、物理环境和实时语音生效的扰动。
- **语音克隆**：用目标说话人的参考音频生成合成语音，攻击语音认证或语音助手。
- **检测难点**：被动检测效果有限，主动水印更有效但可能被绕过。
- **正向利用**：对抗扰动也可用于语音去身份化，在保持语音服务可用性的同时隐藏说话人身份。

## 12. 易混点

- **IoT 不只是“设备联网”**：还包括感知、处理、管理和应用闭环。
- **单层安全不保证整体安全**：每层单独安全，跨层组合仍可能出错。
- **隐私不只是加密传输**：采集什么、存多久、谁能访问、能否再识别同样重要。
- **对抗样本不一定只用于攻击**：也可作为隐私保护工具。

| WSN | Wireless Sensor Network | 无线传感器网络，是 IoT 的重要前身；节点通常有传感器、MCU、无线芯片和电源。 |

| LoRa / NB-IoT | 低功耗物联网通信技术 | LoRa 常在免授权频段，NB-IoT 属蜂窝体系，适合海量低速设备。 |

| DAS/NAS/SAN | 直接/网络/存储区域存储 | 管理层常见存储形态，影响 IoT 数据如何保存、共享和隔离。 |
| IaaS/PaaS/SaaS | 云服务三层 | IaaS 提供基础资源，PaaS 提供平台服务，SaaS 提供应用。 |
| embedding | 嵌入向量 | 语音/图像等输入经模型提取出的特征向量，用于识别或比较。 |
| ASR/DSR/WER/SNR/MOS | 攻击成功率/去身份成功率/词错误率/信噪比/主观质量 | 语音感知安全实验常用指标，分别衡量攻击、隐私、识别错误、可听性和人类感知质量。 |

四层安全对照：

- 感知层：RFID、传感器、摄像头、麦克风；重点是身份认证、抗伪造、抗物理篡改和数据可信。
- 网络层：Wi-Fi、蜂窝、LPWAN、网关；重点是加密、完整性、抗重放、抗 DoS。
- 管理层：云/边缘、数据库、设备管理；重点是访问控制、日志审计、密钥管理、备份恢复和隐私计算。
- 应用层：智能家居、医疗、交通、工业；重点是权限、业务逻辑、用户隐私和安全降级。

PPT 细节补充：

- `Auto-ID Center` 是推动 RFID/EPC 体系的重要研究机构；它把物体自动识别、EPC 编码、网络查询和供应链追踪连起来。
- `EPC system` 是 Electronic Product Code 体系，不只是标签编号，还包括读写器、网络、Object Name Service/解析服务、数据库和应用。
- IoT 的目标常被概括为 `Object equalization`：让物理世界对象像网络资源一样被标识、访问、查询和管理。
- WSN/IoT 节点常形成 `ad-hoc terminal interconnection`，意思是终端能在没有固定基础设施的场景下自组织互联。
- 对抗样本常见公式：`δ = sign(∇x J(θ, x, y))` 表示沿着损失函数对输入的梯度方向添加扰动；实际 FGSM 常写 `x_adv = x + ε sign(∇x J(θ, x, y))`。
- 语音隐私/攻击实验指标：`MCD` 是 Mel-Cepstral Distortion，衡量语音特征失真；`RTF` 是 Real-Time Factor，衡量处理速度是否能实时；`PhoneyTalker` 和 `PhyTalker` 是语音克隆/物理世界攻击案例名。
- `Privacy-Utility Dilemma` 是隐私和可用性的矛盾：隐藏越多个人特征，识别、服务质量或分析准确率可能越低；保留越多数据，隐私风险越高。
- IoT contradictions 可按考试语言写成 high performance vs. low energy、strong security vs. low cost、rich sensing vs. privacy、long lifetime vs. patchability、openness/interoperability vs. attack surface。
- IoT 三个特征要补完整：object equalization、ad-hoc terminal interconnection、`pervasive service intellectualization`。第三个意思是服务无处不在并具备智能化，设备不只是上传数据，还会触发识别、预测、自动控制和个性化服务。

## 历年卷风格练习

1. WSN 的特点有哪些？为什么它是 IoT 的重要基础？
2. 详细阐述 IoT Security Demands，可按 access、communication、data privacy、computing system 分类。
3. 详细阐述 IoT Security Architecture，并说明四层各自保护什么。
4. 为什么语音克隆和对抗样本属于 IoT sensing security 问题？

<details class="self-test-answer">
<summary>参考答案</summary>

1. WSN 节点通常体积小、能量有限、无线通信、分布式部署、自组织、采集物理环境数据。IoT 继承了 WSN 的感知和无线传输能力，并进一步接入云、应用和业务流程。
2. Access security 要防未认证设备/用户接入；communication security 要防窃听、篡改、重放和 DoS；data privacy security 要保护位置、健康、行为等敏感数据；computing system security 要保护固件、OS、云端、边缘和应用免受漏洞和恶意代码影响。
3. Security Architecture 可按 sensing security、transmission security、data/security management、application security 分层。感知层保传感器和 RFID，传输层保网络通信，数据/管理层保云平台和数据库，应用层保业务权限和用户隐私；身份认证、访问控制、加密、审计和密钥管理跨层使用。
4. 语音助手把声音当身份或命令。语音克隆可模仿目标人，对抗样本可让模型误判；攻击发生在传感输入阶段，但后果进入认证、支付、门锁和控制动作，因此属于 sensing security。
